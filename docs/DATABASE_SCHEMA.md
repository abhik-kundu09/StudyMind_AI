# StudyMind AI — Database Schema

Database: **MongoDB** (Motor async driver)
Database name: `studymind`

---

## Critical ID Convention

> Two distinct `_id` conventions coexist in this project. Always check which applies before querying.

| Collection | `_id` type | Notes |
|---|---|---|
| `documents` | **Plain string** (UUID) | Never wrap in `ObjectId()`. Query with raw string. |
| `users` | **Mongo ObjectId** | Accessed as `current_user.id` in routes |
| `chats` | **Mongo ObjectId** | |
| `quizzes` | **Mongo ObjectId** | |
| `attempts` | **Mongo ObjectId** | |
| `study_plans` | **Mongo ObjectId** | |
| `flashcards` | **Mongo ObjectId** | |

**Rule:** `user_id` and `doc_id` fields stored *inside* any collection are always **plain strings**, never ObjectIds — even when the owning collection uses ObjectIds for `_id`.

## Collections

---

### `users`

Stores registered user accounts.

| Field | Type | Notes |
|---|---|---|
| `_id` | ObjectId | Auto-generated by MongoDB |
| `name` | string | Max 100 chars |
| `email` | string | Unique, lowercase |
| `hashed_password` | string | bcrypt hash |
| `created_at` | datetime | UTC |
| `updated_at` | datetime | UTC |

**Indexes:**
- `email` — unique index

---

### `documents`

Stores uploaded PDF metadata. Processing is handled asynchronously by Celery.

| Field | Type | Notes |
|---|---|---|
| `_id` | **string** (UUID) | Plain string — never ObjectId |
| `user_id` | string | Plain string reference to `users._id.str()` |
| `title` | string | Display name |
| `filename` | string | Original filename |
| `status` | string | `PROCESSING` \| `READY` \| `FAILED` |
| `page_count` | integer | Populated after processing |
| `chunk_count` | integer | Number of FAISS-indexed chunks |
| `created_at` | datetime | UTC |
| `updated_at` | datetime | UTC |

**Indexes:**
- `user_id` — for fast per-user listing
- `(user_id, status)` — compound, for filtering by status

**Notes:**
- FAISS index files are stored on disk at `backend/data/faiss_indexes/{user_id}/`, not in MongoDB.
- When a document is deleted, the corresponding FAISS vectors are removed from the index.

### `study_plans`

Stores AI-generated adaptive study schedules.

| Field | Type | Notes |
|---|---|---|
| `_id` | ObjectId | Auto-generated |
| `user_id` | string | Plain string |
| `doc_ids` | list[string] | Plain string document IDs |
| `title` | string | Auto-generated |
| `exam_date` | datetime | UTC |
| `daily_hours` | float | User-specified daily availability |
| `granularity` | string | `HOURLY` \| `DAILY` \| `WEEKLY` |
| `status` | string | `GENERATING` \| `READY` \| `FAILED` |
| `blocks` | list[Block] | Embedded schedule blocks |
| `created_at` | datetime | UTC |

**Granularity selection logic:**
- `exam_date - now < 24h` → `HOURLY`
- `1 day ≤ exam_date - now ≤ 45 days` → `DAILY`
- `exam_date - now > 45 days` → `WEEKLY`

**Block sub-schema (embedded):**

| Field | Type | Notes |
|---|---|---|
| `block_id` | string | UUID string |
| `label` | string | e.g. "Day 1", "Week 3", "9:00 AM" |
| `topic` | string | Study topic for this block |
| `duration_minutes` | integer | |
| `completed` | boolean | Default false |

**Indexes:**
- `user_id` — for plan listing
- `(user_id, created_at)` — compound descending

### `flashcards`

Stores flashcard decks and individual cards with SM-2 spaced repetition state.

| Field | Type | Notes |
|---|---|---|
| `_id` | ObjectId | Auto-generated (this is the deck ID) |
| `user_id` | string | Plain string |
| `doc_id` | string | Plain string |
| `title` | string | Auto-generated |
| `cards` | list[Card] | Embedded — see Card sub-schema |
| `created_at` | datetime | UTC |
| `updated_at` | datetime | UTC |

**Card sub-schema (embedded):**

| Field | Type | Notes |
|---|---|---|
| `card_id` | string | UUID string |
| `front` | string | Question / prompt side |
| `back` | string | Answer / explanation side |
| `due_date` | datetime | Next review date (SM-2) |
| `interval` | integer | Current inter-review interval in days |
| `ease_factor` | float | SM-2 ease factor (default 2.5) |
| `repetitions` | integer | Number of successful reviews |

**SM-2 Algorithm summary:**
- Quality 0–1: reset interval to 1, ease decreases
- Quality 2: repeat interval, ease decreases slightly
- Quality 3–5: interval increases by `interval * ease_factor`, ease adjusts
- Ease factor floor: 1.3
- New cards start with `interval=1, ease_factor=2.5, repetitions=0`

**Indexes:**
- `user_id` — for deck listing
- `(user_id, doc_id)` — compound, for doc-specific deck lookup
- `(user_id, "cards.due_date")` — for due card queries

## Progress — No Separate Collection

The `progress` feature has **no dedicated MongoDB collection**. All progress stats are computed on-demand in `progress_service.py` by aggregating across:

- `attempts` — quiz scores, quiz count, weekly activity
- `study_plans` — completed blocks, study days
- `flashcards` — review sessions (updated_at tracking)

**Key fields returned by `/progress/summary`:**

| Field | Source collection |
|---|---|
| `streak` | `attempts` + `study_plans` by date |
| `total_quizzes` | `attempts` count |
| `study_days` | `study_plans` block completion dates |
| `flashcard_sessions` | `flashcards` updated_at dates |
| `avg_quiz_score` | `attempts` percentage average |
| `weekly_activity` | All three collections, last 7 days |

## Datetime Handling

MongoDB may store datetime fields as native Python `datetime` objects or as ISO strings depending on insertion path. Always use the `_to_dt()` coercion helper in `progress_service.py` before calling `.tzinfo`, `.strftime()`, or any datetime arithmetic.

```python
def _to_dt(value) -> datetime:
    if isinstance(value, datetime):
        return value
    return datetime.fromisoformat(str(value))
```
